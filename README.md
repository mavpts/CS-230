# CS-230
SNHU CS 230
Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
The Gaming Room has a program named Draw It or Lose It. Loosely based on the 1980s television game Win, Lose or Draw. The goal of The Gaming Room was to transform their application from a an android only game to a global cross platform game using java. 
What did you do particularly well in developing this documentation?
For me the documentation was really straight forward. It was simply a template that told you exactly what was needed where.
What about the process of working through a design document did you find helpful when developing the code?
The document created a backlog of sorts. It showed exactly what the client wanted without a need for improvisation. This kept fluff programming to a minimum by pinpointing exactly what needed done.
If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
One of the corrections I received was that my opinions were biased. And the instructor was correct. I’ve had some bad experience with IOS and recently have become very irritated about windows and their way of forcing people to upgrade hardware to continue to receive updates. In that sense, I should have been more open to software other than just Linux/Android.
How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
For this project, we already had an established product. We knew what the customer’s ultimate goal was for it. Unlike making a completely fresh program, we already had a working example. So unlike a fresh program where you would constantly want to check user stories and feedback, here we can put a higher priority on porting the application. 
How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
For this application we used a java for the server and client. This makes the application great for cross platform since almost every modern client device supports java. We used the Singleton pattern which allowed us to prevent multiple instances from appearing on the same device. This will be useful later when developing other cross platform devices as it will make cheating harder.
